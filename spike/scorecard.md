      (1) over-fetches the whole reply tree → noisy + slow (up to 37s) + UNBOUNDED cost (exhausted
      the credit balance mid-corpus → HTTP 402 on the last 3), and (2) still misses X-article bodies
      (returns reader reactions, e.g. aiedge_=1099 / zodchii=1233 chars vs Jina's 12.9k / 13k).
    - **Firecrawl → RULED OUT, removed from harness** (untested — no key — but moot since Jina
      already wins on cost, speed, and coverage; no reason to spend evaluating it further).
- Projected weekly cost (15–20 items): **~$0 for fetch** (Jina free tier + YouTube transcript free).
  Only real cost is the Claude ranking call in Phase 1. (Jina paid tier is a few $/mo if free limits bite.)
- Does the two-pass model collapse for X (full thread cheap in one call)? **Yes — and for everything.**
  Jina returns full content in one free ~460ms call, so the cost pressure that motivated two-pass
  (don't pay to fully extract items that won't survive ranking) disappears. Full-extract everything up
  front for $0; rank on full text. Drop the two-pass design for Phase 1/2.

### NEW finding from the eyeball pass: Jina loses links on "tweet + linked article" URLs — TWO
### distinct failure modes, of different severity. Verified against `out/2060468027703849017__x-jina.md`
### (the 0xwhrrari "prompting 101" tweet), which has both an inline link AND an article-preview card.

**1. Inline `t.co` links in the tweet body → present but unresolved (recoverable).**
The tweet text is captured verbatim including `…Now that opus 4.8 https://t.co/K8UN1sM5Ip` (line 14)
— an opaque shortened string, not rendered as a clickable link, destination not resolved. Confirmed
in 8/14 captures. Checked what these resolve to (one cheap `curl -L`, full redirect chain, $0):
  - `K8UN1sM5Ip`, `tmapFcMvdo`, `lRLjUVn6Zm`, `XLdBtTfbHV` → all resolve to **the tweet's own attached
    video** (`x.com/.../video/1`) — self-referencing media, not external linked content.
  - `9aNlqZ7hIS` → resolves to a genuine **external article** (`ruben.substack.com/p/youre-just-a-text-file`).
  → Fixable: resolve via one HEAD redirect, discard self-referencing targets, queue genuine externals.

**2. "X article" preview-card links → DROPPED ENTIRELY (NOT recoverable from the capture).**
This is the more serious gap, surfaced when the user checked the live tweet: the card visible at the
bottom of `2060468027703849017` links to **`x.com/rubenhassid/status/2050787472666399042`** — a
completely separate X-article by a different author. That URL appears **nowhere** in the 4,242-char
Jina capture (`grep` confirms zero matches). All that survives is the cover image and title text:
  `![Image 3: Article cover image](...HJV5FdNWwAM1OJD...)` / `Article` / `How I set up Claude to
  actually get work done` (lines 63–67).
There is no shortlink, no `href`, nothing to resolve — Jina simply never emits the card's destination
URL into markdown. Several `urls.json` entries are explicitly typed "X article" — if this pattern holds,
**linked X-articles surfaced via preview cards are structurally unreachable through Jina**, a real
content-coverage gap, not a cosmetic noise issue.

**Implications for Phase 2:**
  - Inline `t.co` links: small, well-understood fix (resolve → filter self-referencing → queue externals).
    Lower priority — most resolve to self-referencing media anyway (4/5 sampled).
  - Article-preview-card links: **RESOLVED by capture-time decision (2026-06-07)** — user will take on
    responsibility for choosing the right URL to capture: push the **linked X-article's URL** when a
    card is present, or the **tweet's own URL** when it isn't. This sidesteps the unrecoverable-link
    problem entirely — no fetch-side resolution code needed, since the chosen URL is itself an `x.com`
    URL and routes through the already-locked `x:jina` adapter either way. Zero new adapter work.
    (Trade-off: shifts a small judgment call onto the human at capture time — acceptable given the
    user explicitly chose to own it, and it's a one-glance decision: "is there a card at the bottom?")
